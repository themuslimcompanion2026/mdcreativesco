## Problem
The deploy is completing, but the live site crashes at request time and falls back to the custom 500 page. The logs strongly point to a request-time SSR failure, not a build/output problem:
- `dist/pages`, `_worker.js`, and `_routes.json` are valid
- upload succeeds on Cloudflare Pages
- Cloudflare reports **no environment variables found** during build
- shared SSR-rendered UI currently pulls backend data through client-side hooks/components that can fail on the server when backend env is missing or unavailable

## Plan

### 1) Remove SSR crash points from the public layout and homepage
Refactor the public data access used by shared SSR paths so the site can render safely on Cloudflare:
- replace direct backend-dependent reads in shared public hooks/components with request-safe access patterns
- make the global shell (`Header`, `Footer`, `BrandLogo`, favicon, homepage sections) render with safe defaults/fallbacks instead of throwing
- ensure missing backend config does **not** take down `/`, `/about`, `/services`, `/portfolio`, `/pricing`, `/contact`, or `/book`

### 2) Harden backend/env access for Cloudflare Pages
Add a small app-level env guard layer around public data usage:
- detect whether required public/runtime backend vars are available at request time
- avoid triggering backend queries during SSR when config is missing
- surface degraded-but-renderable behavior on public pages instead of a 500
- keep admin/payment/invoice routes explicit when backend config is required

### 3) Improve runtime diagnostics so the real cause is visible
Strengthen deployment diagnostics and health reporting:
- make `/api/health` clearly show which build-time and runtime variables are missing
- improve request/error logging so request-time SSR failures are easier to trace in Cloudflare logs
- make diagnostics distinguish between “worker attached” and “backend not configured”

### 4) Update deployment guidance for Cloudflare Pages
Tighten the deployment docs/checklist so the Pages project is configured correctly:
- Production + Preview build vars: `VITE_*`
- Production + Preview runtime secrets: backend/service keys and payment/webhook secrets
- clarify that a successful upload is not enough; missing Pages variables will still break live SSR requests

### 5) Validate end-to-end after the fix
After implementation, verify the exact failure mode is gone:
- production build succeeds
- Pages output still contains `_worker.js` and `_routes.json`
- no stray `wrangler.json`
- homepage loads instead of showing the fallback 500 page
- `/api/health` reports useful diagnostics
- admin/payment/invoice flows fail gracefully when config is missing and work correctly when config is present

## Technical details
- Most likely crash surface: `src/hooks/useSiteData.ts` and the shared components that call it from the SSR tree (`Header`, `Footer`, `BrandLogo`, `DynamicFavicon`, `TrustStats`, `Pricing`, `Portfolio`)
- I will **not** edit autogenerated backend integration files
- I’ll fix this in app code by making shared SSR paths backend-safe and by improving request-time diagnostics
- One external step will still be required for full production functionality: Cloudflare Pages must have the required env vars/secrets configured in the dashboard for both Preview and Production. Code can prevent the crash and expose the problem clearly, but it cannot create dashboard secrets from the repo itself.
